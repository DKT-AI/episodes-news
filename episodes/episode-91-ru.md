### 1. Before You Migrate: Five Surprising Ingress-NGINX Behaviors You Need to Know | Kubernetes
В марте 2026 года Kubernetes официально выводит из эксплуатации Ingress-NGINX, и команды по всему миру готовятся к миграции на Gateway API. Статья описывает пять неочевидных поведений контроллера — от глобального применения regex-аннотаций до автоматической нормализации URL — которые могут сломать маршрутизацию, если не учесть их заранее.
**Ссылка:** https://kubernetes.io/blog/2026/02/27/ingress-nginx-before-you-migrate/
**Тезисы для обсуждения:**
- Насколько эти «сюрпризы» действительно неочевидны для команд, которые давно работают с Ingress-NGINX, и были ли у вас случаи, когда одно из этих поведений приводило к инциденту в продакшене?
- Аннотация `use-regex` применяется глобально ко всем путям хоста — это классический пример «удобной фичи», которая становится источником трудноуловимых багов при масштабировании. Как Gateway API решает эту проблему архитектурно?
- Миграция на Gateway API — это не просто замена YAML-манифестов: меняется модель безопасности, разделение ответственности между инфраструктурой и приложениями. Готовы ли большинство команд к этому концептуальному сдвигу?

### 2. Evaluating AGENTS.md: Are Repository-Level Context Files Helpful for Coding Agents? | HackerNoon
Концепция AGENTS.md предлагает хранить в репозитории специальный файл с инструкциями и контекстом для AI-агентов — по аналогии с тем, как README объясняет проект людям. Статья ставит вопрос: действительно ли такие файлы помогают агентам лучше следовать инструкциям, понимать структуру проекта и не злоупотреблять вызовами инструментов?
**Ссылка:** https://hackernoon.com/evaluating-agentsmd-are-repository-level-context-files-helpful-for-coding-agents
**Тезисы для обсуждения:**
- AGENTS.md — это попытка формализовать «контракт» между разработчиком и AI-агентом на уровне репозитория. Насколько этот подход масштабируется в реальных проектах с несколькими командами и сотнями сервисов?
- Если агент игнорирует или неправильно интерпретирует инструкции из AGENTS.md, кто несёт ответственность за результат — разработчик, написавший файл, или инструмент? Как это влияет на доверие к AI-агентам в CI/CD-пайплайнах?
- Можно ли рассматривать AGENTS.md как эволюцию Infrastructure as Code — только теперь мы описываем не инфраструктуру, а намерения и ограничения для автономных агентов?

### 3. Kubernetes PKI & Kubelet Credential Abuse: From Popping a Pod to Owning the Cluster | Medium
Статья демонстрирует полную цепочку атаки: получив доступ к файловой системе узла, злоумышленник крадёт `ca.key` из `/etc/kubernetes/pki`, подписывает собственный сертификат с группой `system:masters` и получает права cluster-admin. Это не теоретический сценарий — это реальный вектор, актуальный для кластеров с недостаточно жёсткими правами на PKI-директории.
**Ссылка:** https://medium.com/@deepanshu_khanna/kubernetes-pki-kubelet-credential-abuse-from-popping-a-pod-to-owning-the-cluster-263a70cc6e83
**Тезисы для обсуждения:**
- Цепочка атаки начинается с «всего лишь» доступа к файловой системе узла — насколько реалистичен этот начальный вектор в типичных production-кластерах, и какие конфигурационные ошибки чаще всего к нему приводят?
- Использование HSM для защиты CA-ключей — правильная рекомендация, но насколько она применима для большинства команд с точки зрения стоимости и операционной сложности? Какие более доступные меры дают реальный эффект?
- Мониторинг аномальных вызовов API — последний рубеж обороны в описанном сценарии. Какие конкретные сигналы и инструменты (Falco, audit logs) позволяют обнаружить подобную атаку до того, как злоумышленник закрепится в кластере?
